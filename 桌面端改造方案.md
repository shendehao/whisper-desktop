# Whisper Desktop 桌面端改造方案

> 版本：v2.0 → v3.0  
> 目标：高端优雅、动效流畅、暗黑模式无色差、更新闭环可用、超管权限清晰。

---

## 一、当前问题盘点

### 1. UI 层面（外观）
- 整体只是 Element Plus 默认样式 + Tailwind 拼接，**没有设计语言**，廉价感重。
- `el-card`、`el-table`、`el-menu` 在暗黑模式下颜色不联动：卡片始终是白底，表格表头始终是浅灰，与左侧深色侧边栏脱节（见用户反馈图 6）。
- 数据面板数字颜色过于跳跃（蓝/绿/紫/橙/红/青/灰），像随机配色，**不优雅**。
- 侧边栏菜单文字在暗黑模式下是固定 `#9CA3AF`，悬停/选中无渐变与左侧高亮条。
- 没有任何过渡动画：页面切换硬切、卡片加载无淡入、数字无滚动动画、按钮无微交互。
- 颜色反转问题：`filter: invert(1)` 用在 logo 上，主题切换时不会平滑过渡；`text-gray-500` 这类 Tailwind 固定色没有 dark: 变体。

### 2. 功能层面
- **更新流程残缺**：后端 `/api/desktop/check-update` 已就绪，但客户端**没有调用、没有弹窗、没有下载、没有提示**。推送了等于没推送。
- 没有「检查更新」按钮（设置页）。
- 没有强制更新阻塞逻辑。
- 没有版本号在标题栏 / 状态栏的展示。

### 3. 权限/数据层面
- 超级管理员在产品定义上是「一个普通租户 + 一些运维与全局管理功能」，**业务数据必须与其它租户完全隔离**（卡密、软件、云变量、公告、子账号、黑名单只能看自己拥有的）。
- 当前侧边栏把超管的"个人业务菜单"和"全局运维菜单"混在一起，新人会以为超管能看到所有租户的卡密，**视觉上没有分组**。

## 三、技术栈选型

| 用途 | 选型 | 理由 |
| --- | --- | --- |
| 组件库 | **保留 Element Plus**（不换） | 沉没成本太大，重点是做主题覆盖与外壳重构 |
| 动画 | **@vueuse/motion** + 原生 CSS transition | 体积小，声明式 `v-motion`，与 Vue 3 SSR-free 完美兼容 |
| 数字滚动 | **vue-countup-v3** | 仪表盘数字滚动到位 |
| 图标 | **@iconify/vue**（lucide / tabler 集合） | Element Plus 自带图标太少且风格陈旧 |
| CSS | 保留 Tailwind，但所有颜色走 CSS 变量 | 解决暗黑模式色差 |
| 版本比较 | 内置实现 `parseVer()` | 不引依赖 |

```bash
npm i @vueuse/motion vue-countup-v3 @iconify/vue
```

## 七、更新闭环（重点：补完）

### 7.1 composable `src/composables/useUpdateCheck.ts`

```ts
import { ref } from "vue";
import http from "@/api";

export interface UpdateInfo {
  has_update: boolean;
  force_update: boolean;
  latest_version: string | null;
  current_version: string;
  update_url: string;
  update_content: string;
  min_version: string;
  pushed_at: string | null;
}

const CURRENT_VERSION = "2.0.0";  // 与 tauri.conf.json 同步，可通过 import.meta.env.VITE_APP_VERSION 注入

export const updateInfo = ref<UpdateInfo | null>(null);
export const checking = ref(false);
export const lastError = ref<string>("");

export async function checkUpdate(silent = true) {
  checking.value = true;
  lastError.value = "";
  try {
    const res = await http.get<UpdateInfo>("/api/desktop/check-update", {
      params: { current_version: CURRENT_VERSION },
    });
    updateInfo.value = res.data;
    return res.data;
  } catch (e: any) {
    lastError.value = e?.message || "检查更新失败";
    if (!silent) throw e;
    return null;
  } finally {
    checking.value = false;
  }
}

export function startAutoCheck(intervalMs = 30 * 60 * 1000) {
  checkUpdate(true);
  setInterval(() => checkUpdate(true), intervalMs);
}
```

### 7.2 弹窗组件 `src/components/UpdatePrompt.vue`

行为矩阵：

| 场景 | 弹窗类型 | 用户操作 |
| --- | --- | --- |
| `has_update=false` | 不显示 | — |
| `has_update=true, force_update=false` | 右下角悬浮卡片，可关闭，可"稍后提醒" | 立即更新 / 稍后 / 跳过此版本 |
| `has_update=true, force_update=true` | 全屏遮罩 modal，不可关闭 | 仅"立即更新" |

弹窗内容：
- 标题：发现新版本 `v{latest_version}`
- 副标题：当前 `v{current_version}` → `v{latest_version}`
- 内容：`update_content` 多行渲染（保留换行）
- 按钮：立即更新（点击后用 Tauri `shell.open(update_url)` 在系统浏览器下载安装包，或集成 tauri-plugin-updater 做静默升级）

集成位置：`Layout.vue` 根节点放 `<UpdatePrompt />`，登录成功后立即触发 `startAutoCheck()`。

### 7.3 强制更新阻塞

`Layout.vue` 中：
```vue
<div v-if="updateInfo?.force_update" class="force-update-blocker">
  <UpdatePrompt force />
</div>
<router-view v-else />
```

强制更新时，路由不渲染，用户只能更新或退出。

### 7.4 跳过此版本

写入 `localStorage.setItem("whisper_skip_ver", latest_version)`。  
下次检查时若 `latest_version === skipped && !force_update`，则不弹窗。

### 7.5 设置页扩展

`Settings.vue`：
- 当前版本：v2.0.0
- 「检查更新」按钮 → `checkUpdate(false)`，无更新时 toast「已是最新版本」，有更新则弹窗。
- 显示上次检查时间。

## 八、超管权限与数据隔离

### 8.1 产品定义（写入文档）

> 超级管理员 = 「一个普通租户」 + 「全局运维与租户管理功能」。  
> 业务数据（卡密、软件、云变量、公告、黑名单、子账号）**严格按 `owner_id == 自身用户 ID** 隔离，超管也只能看见自己作为租户创建的数据。  
> 全局运维菜单（租户管理、系统监控、全局日志、系统设置）才能看到全平台聚合数据。

### 8.2 菜单分组

```ts
const businessGroup = [dashboard, software, cards, cloudVars, announcements, blacklist, deviceBlacklist, subAccounts, logs];
const opsGroup = [users /* 租户管理 */, monitor, globalLogs, settings];  // 仅 role === "superadmin" 可见
```

侧边栏渲染时按 group 分组显示，运维组顶部带分隔线和「运维管理」灰色小标题。

### 8.3 接口侧校验（已具备，回归确认）

- 所有业务接口必须经过 `_effective_owner_id(request)` 过滤，**不允许超管 bypass**。  
  已修复点：`app/admin_panel/views/lua_script.py` 的 `ajax_lookup` / `get_query`。  
  **需复盘**：`desktop_api/cards.py`、`desktop_api/software.py`、`desktop_api/cloud_vars.py`、`desktop_api/announcements.py`、`desktop_api/blacklist.py`、`desktop_api/sub_accounts.py` 是否都在用 `owner_id == current_user.id` 而不是 `if is_superuser: skip filter`。
